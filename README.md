# SAS Modernization MVP ‚Äî Synthetic Test Bench

A small, self-contained SAS codebase plus ground-truth outputs, designed
to exercise a Claude Code-driven modernization pipeline end-to-end.

## What's in here

```
.
‚îú‚îÄ‚îÄ CLAUDE.md                 <- the operating instructions for Claude Code
‚îú‚îÄ‚îÄ solution_design.md        <- the team's pipeline spec (5-phase architecture)
‚îú‚îÄ‚îÄ README.md                 <- this file (for human readers, not Claude Code)
‚îú‚îÄ‚îÄ sas_codebase/             <- the SAS source: 5 programs, 1 macro file, 1 format catalog, docs
‚îú‚îÄ‚îÄ input_data/               <- synthetic raw inputs (CSV)
‚îú‚îÄ‚îÄ ground_truth/             <- "what the SAS would produce" ‚Äî the validation target
‚îî‚îÄ‚îÄ generate_data_and_truth.py  <- regenerates input + ground-truth from seed=42
```

## Domain

A clinical-trials SDTM ‚Üí ADaM pipeline for a fictitious Phase 2 study
(`CTX-2024-001`). This is the standard pharma SAS workload ‚Äî realistic
complexity without inventing a domain model.

## What the SAS code does

Five programs, dependencies as below:

```
01_clean_dm.sas      02_clean_ae.sas
       \                 /
        v               v
   03_derive_adsl.sas  (sets &TRT_START_DT global)
              |
              v
        04_derive_adae.sas (uses &TRT_START_DT)
              |
              v
       05_summary_safety.sas
```

Outputs: `DM_CLEAN`, `AE_CLEAN`, `ADSL`, `ADAE`, `AE_SUMMARY`, `AE_INCIDENCE`.

## Planted complexity (this is the demo's value)

The point of the synthetic codebase is to exercise things that distinguish
a real modernization pipeline from a regex translator. Specifically:

| # | What's planted                              | Where                                  | Why it matters                                                                 |
|---|---------------------------------------------|----------------------------------------|--------------------------------------------------------------------------------|
| 1 | Cross-program global macro side effect      | `03_derive_adsl.sas` sets `&TRT_START_DT`, `04_derive_adae.sas` reads it via `%is_treatment_emergent` | Cannot be detected by single-program analysis; needs the cross-program DFG     |
| 2 | Implicit type cast in PROC SQL join         | `03_derive_adsl.sas` joins `DM.SITEID` (char `'01'`) on `SITE_LOOKUP.SITE_ID` (num `1`) | SAS coerces silently; PySpark will not. Must be flagged, not guessed           |
| 3 | Severity codes that fall through the if-then chain | `02_clean_ae.sas` standardizes `MILD/MOD/SEV/1/2/3` but `GRADE 1` etc. become blank | Tests "missing vs null vs blank" handling; doc explicitly notes it (SP-184)    |
| 4 | DM duplicate records resolved by sort+first | `01_clean_dm.sas`                       | BY-group semantics; LAST/FIRST. behavior must be preserved                     |
| 5 | LAST. on max-severity per subject           | `02_clean_ae.sas`                       | Same                                                                           |
| 6 | Missing AGE derived from BRTHDTC            | `01_clean_dm.sas`                       | Conditional derivation; depends on field-missingness check                     |
| 7 | Drop on missing date / missing term         | `02_clean_ae.sas`                       | Implicit row-level filter, easy to lose in translation                         |
| 8 | Spec/code disagreement: "first dose" vs "randomization date" | functional_spec.md ¬ß6.3 vs implementation | Documentation NLP must catch the gap, not assume code matches doc              |
| 9 | Format catalog with `other` fallthrough     | `formats.sas`                           | Tests that the graph captures the format catalog as a first-class entity       |
| 10| Macro depending on global, with no parameter | `%is_treatment_emergent` in `util_macros.sas` | The classic "looks pure but isn't" pattern                                     |

A successful run produces an `ambiguity_log.md` that names items 1, 2, 3,
and 8 explicitly. Items 4‚Äì7 should be handled correctly (and verified by
ground-truth diff). Item 9 should appear as a node in the knowledge graph.

## Ground truth

`ground_truth/*.csv` are computed by `generate_data_and_truth.py`, which
mirrors the SAS logic in Python (with the same seed and the same
implicit-cast behavior, deliberately). The PySpark generated by Claude
Code should diff cleanly against these files.

To regenerate:
```bash
python3 generate_data_and_truth.py
```

## How to use this with Claude Code

1. `cd` into the unzipped directory.
2. Start `claude code`. It will read `CLAUDE.md` automatically.
3. Claude Code will **enter plan mode and ask you 7 clarifying questions**
   (target language, KG backend, parser approach, ambiguity policy, output
   format, validation strictness, test framework). Answer them.
4. Approve the plan when prompted. Claude Code then executes phases 1‚Äì5.
5. Watch the `build/` directory fill up phase by phase. The single most
   important output is `build/SOLUTION.md`, which contains 9 named
   sections including a dedicated dependency-graph section with three
   Mermaid diagrams (program-level, dataset-level, column-level lineage).
6. Final check: `diff <(sort build/target/output/ae_summary.csv) <(sort ground_truth/ae_summary.csv)` should be empty.

If the pipeline shortcuts to "just translate the SAS", the demo has
failed. The whole point is the AST ‚Üí graph ‚Üí spec ‚Üí codegen separation,
the dependency graph being explicit, and the ambiguity log being non-empty.

### What the final SOLUTION.md should contain

Nine sections, in order:
1. Executive summary
2. Codebase inventory
3. **Dependency graph** (program DAG, dataset lineage, column lineage ‚Äî all Mermaid)
4. Knowledge graph schema and statistics
5. Ambiguity register (table)
6. Generated artifacts inventory
7. Validation results (diff vs ground truth)
8. Coverage report
9. Recommendations for production rollout

A reviewer should be able to read this single document end-to-end and
understand the modernization without opening any SAS or generated code.

## Caveats (be honest in the demo)

- This is a synthetic codebase. It does not contain `%sysfunc`, dynamic
  `&&var`, PROC SQL passthrough to a remote DB, or SAS macros that
  generate code at runtime. Real client codebases will have these and
  the pipeline will need extensions.
- 20 subjects, 58 AEs. The data is small by design ‚Äî the MVP is about
  approach validity, not scale.
- The ground truth was produced by *Python mirroring the SAS logic*, not
  by an actual SAS compiler. If you have access to a SAS environment and
  want to regenerate ground truth from real SAS, the input CSVs are the
  same.
